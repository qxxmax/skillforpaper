# Interaction Protocol

Use this reference when the task involves iterative research writing, poster
design, PPT design, paper drafting, or skill iteration.

## Why Record Interaction

Research communication is not a one-shot extraction problem.  The user often
knows the intended emphasis before the agent does.  Record the interaction so
future agents and users can see:

- what the user asked for
- what the agent produced
- what the user said was wrong or missing
- how the artifact changed
- what remains unresolved

## Feedback To Action

| User Feedback | Likely Diagnosis | Required Action |
|---|---|---|
| "The focus is unclear." | The thesis exists but is not visually or structurally dominant. | Promote the thesis to the title, first paragraph, or largest visual object. |
| "It is not complete." | The public artifact is being judged as if it were the source ledger. | Expand the source map/evidence ledger, then choose what belongs in the poster/PPT/paper. |
| "You did not show the formula/claim." | The mechanism is described in prose but not embodied visually. | Create formula cards and a problem-method-formula-evidence-conclusion strip. |
| "How should people interact with this?" | The workflow is not user-teachable. | Add an interaction log, reusable prompts, and a checkpoint question. |
| "Do not edit the writing yet." | The user wants diagnosis before prose. | Produce maps, trees, inventories, and visual candidates first. |
| "This is still too simple." | A one-page overview is acting as the whole explanation. | Promote the artifact into a multi-board diagnostic pack where each board explains one block. |
| "Can we keep looking for literature?" | The field position is changing as the argument changes. | Run the iterative literature loop and feed the gap ledger back into the paper, proposal, or slides. |
| "Start polishing the proposal." | The user wants funder/reviewer-facing argument, not only a manuscript summary. | Build the ask, gap, aims, feasibility, risks, and reviewer-objection ledger before line editing. |
| "Make slides." | The user needs a spoken visual story, not a compressed paper. | Build a deck spine, slide inventory, visual anchors, and backup plan before creating slide files. |

## Checkpoint Questions

Ask these after the first diagnostic pass:

1. Is this the one-sentence thesis you want readers to remember?
2. Is the source map complete enough, or did I collapse important sections?
3. Which formulas or concepts must become visuals?
4. Which audience is this for: reviewers, poster judges, collaborators, students, or funders?
5. Should the next artifact be a paper outline, slide deck, poster, or figure pack?
6. Should we run no, quick, full, or monitor-mode literature search before the next artifact?

## Case Pattern From NHMC Book

First pass:

- User requested a poster but explicitly wanted four diagnostic products first.
- Agent produced a v1 diagnostic poster with chapter blocks, argument tree,
  formula inventory, and visual candidates.

User correction:

- "重点没展示清楚": the central thesis was not dominant enough.
- "没展示全?": the chapter map was too compressed.
- "记录我们这样交互的过程": the skill needs a visible collaboration protocol.

Required repair:

- Create a full source map with every chapter and appendix.
- Redesign the poster around one large thesis.
- Add an interaction log and reusable prompts to the skill.
- If the revised one-page poster still feels too simple, split it into an
  expanded board pack: overview, chapter maps, argument tree, formula inventory,
  evidence gallery, and interaction protocol.

This pattern should generalize to other books, papers, and datasets.
