# Poster Workflow

Use poster mode when the user wants a poster, a Paper2Poster-style extraction,
a formula/claim visual, or a diagnostic poster from a paper, book, draft, data
folder, or source tree.

## Required First Pass

Do not begin with final wording.  First produce:

1. Chapter/project map.
2. Argument tree.
3. Formula/concept inventory.
4. Display-figure candidates.

The first pass is successful only if the user can answer:

- "Is this the main point readers should remember?"
- "Is anything important missing from the source map?"
- "Which proof objects should carry the poster?"

## Poster Versus Source Map

The poster is not the full ledger.  Use this split:

- Source map: complete chapter/section/evidence table.
- Poster: one dominant thesis, one mechanism, three to five proof objects.
- Supplement or deck: extra chapters, appendices, formulas, and diagnostics.

When a user says "not complete," expand the source map first.  Then redesign the
poster to signal that a complete source map exists, for example with a compact
chapter ladder or coverage strip.

When a user says the result is still too simple, do not keep crowding the same
poster.  Create an expanded multi-board diagnostic pack first:

- Overview / central thesis.
- Full chapter or project map.
- Argument tree.
- Formula or concept inventory.
- Evidence gallery and display candidates.
- Interaction protocol and next checkpoint questions.

Then choose one board as the public poster spine or convert the pack into a PPT.

## Focus Test

Before exporting, check:

- The thesis is visible in the first three seconds.
- The method mechanism is a diagram, not only prose.
- Each evidence panel has one claim and one source anchor.
- Boundary language is visible but not defensive.
- The poster does not hide failures, negative audits, or reserved slots.

## Scientific Poster Spine

Recommended layout:

```text
title / thesis
problem -> method -> core formula -> diagnostic gates -> conclusion
full-source-map signal
proof objects
boundary / takeaway
```

For formula-heavy work, use formula cards instead of paragraph boxes.

After the source map and formula inventory are accepted, create a layout
specification before rendering.  Read
`references/16_poster_layout_prompt_protocol.md` when the task needs a
portrait/landscape decision, section grid, visual hierarchy, image-generation
prompt, or camera-ready layout plan.
Read `references/17_camera_ready_poster_rebuild.md` when a rough poster,
AI-generated PNG, screenshot, or prototype must be rebuilt as editable PPT,
Illustrator/Inkscape, LaTeX, or deterministic-script output with checked
formulas, verified numbers, and vector-ready plots.

## Versioning

Save each major pass with a version suffix and a short reason:

- `v1_diagnostic`: first extraction.
- `v2_full_map`: completeness repair.
- `v3_focus`: thesis and visual hierarchy repair.
- `v4_focus`: one-page front-door poster.
- `v5_panel_pack`: multi-board explanation when one page is too simple.
- `v6_export`: final poster/PPT/PDF export.

Record user feedback that caused each version.
